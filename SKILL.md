# 轮扁 | 得之于手而应于心

> 做一遍，它就学会了。把你手头的操作变成可复用的 Skill——不用写一行 prompt。
> 从行为捕捉到多平台封装，一条龙全自动。
>
> **典故**：出自《庄子·天道》——"轮扁斫轮于堂下……得之于手而应于心，口不能言，有数存焉于其间。臣不能以喻臣之子，臣之子亦不能受之于臣。"轮扁是春秋时期齐国造车轮的巧匠，技艺已入化境却不善言辞。齐桓公问他为何不传手艺给儿子，他说真正的技艺只能从实践中悟出，无法通过言语传递。这正如本 Skill 的精髓：用户不必会写 prompt，只需亲手做一遍，Skill 便从操作中学会。

---

## 设计哲学

三个核心理念支撑整个锻造过程：

1. **从行为到意图**：用户不需要会写 prompt。通过观察鼠标轨迹、键盘输入、屏幕焦点区域和操作结果，逆向推断真实意图，对话澄清后锁定目标。
2. **锻造即进化**：生成的 Skill 不是一次性产物。内置 Darwin 棘轮机制——评分 → 改进 → 实测 → 保留或回滚，Skill 会自我生长。
3. **出生即发布**：每个 Skill 产出时已封装为跨平台标准包，含 README、showcase、测试集、依赖声明，可直接上传 GitHub / ClawHub / skills.sh / UUmit / Skill 商城。

## Phase 0：接活——判断锻造路径

### 输入类型自动识别

收到用户输入后，首先判断属于以下哪种，走对应路径：

| 输入特征 | 锻造路径 | 触发词示例 |
|---------|---------|-----------|
| 无文字 / 极短模糊指令（"帮我做个那个""搞一个刚才那样的"） | **行为捕捉路径** → Phase 1-A | "那个""这样的""像刚才那样" |
| 用户明确说"看我的操作""跟着我做""录我的步骤" | **行为捕捉路径** → Phase 1-A | "看我的操作""跟着我做""录步骤" |
| 自然语言描述了 Skill 功能/目标 | **直接生成路径** → Phase 1-B | "做一个自动整理桌面的skill""我想要一个..." |
| 给出了已有 SKILL.md / 目录 / GitHub 链接 | **打磨路径** → 鲁班验料流程 | "打磨这个""优化这个""帮我改改" |

### 不能确定时

使用 `ask_user` 给出两个选项：
- "按你描述的，我直接生成 Skill"
- "你来操作，我观察你的行为来理解需求"

## Phase 1-A：识意——行为捕捉理解

> 适用于不会写 prompt / 无法准确描述需求的用户。

### 行为捕捉启动

告知用户："我会观察你的操作。请开始操作，完成后告诉我'好了'。"

### 可捕捉的行为维度

| 维度 | 采集内容 | 推断意图 |
|------|---------|---------|
| 鼠标轨迹 | 点击位置、悬停区域、拖拽路径、右键菜单 | 操作目标是什么控件/文件/按钮 |
| 键盘输入 | 按键序列、快捷键组合、输入文本 | 触发什么命令、输入什么数据 |
| 屏幕焦点 | 活动窗口标题、UI 元素识别、截图分析 | 在哪个应用/页面操作 |
| 操作节奏 | 点击间隔、停顿位置、重复模式 | 熟练度、犹豫点（可能是模糊需求点） |
| 结果反馈 | 操作后的屏幕变化、文件变化、输出内容 | 期望的最终状态 |

### 行为维度 → 工具实现映射

每个行为维度依托当前 Agent 运行时可用的工具来实现采集和推断：

| 行为维度 | 采集方式 | 对应工具 | 推断逻辑 |
|---------|---------|---------|---------|
| 鼠标轨迹 | 截取用户操作前后的屏幕变化 | `analyze_image` 对比前后截图 | 点击了什么控件/按钮/文件；拖拽的起点和终点 |
| 键盘输入 | 监听用户输入文本和快捷键序列 | 用户对话中的文本输入 + 上下文推断 | 触发什么命令、输入什么数据、用了什么快捷键 |
| 屏幕焦点 | 截取活动窗口区域 | `shell_executor` 获取活动窗口标题 + `analyze_image` 识别 UI | 在哪个应用/页面操作、窗口标题提供上下文线索 |
| 操作节奏 | 记录操作之间的描述和时间间隔 | 对话轮的间隔分析 | 停顿点 = 模糊需求点；重复模式 = 批量操作意图 |
| 结果反馈 | 截取操作完成后的屏幕状态 | `analyze_image` 截取结果态 → 对比操作前 | 最终文件列表、输出、界面变化 → 推断期望状态 |

**行为捕捉的实际执行流程**：

```
1. 告知用户："请开始操作，完成后告诉我'好了'。"
2. 用户操作期间：
   - 每完成一步关键操作后，提示用户：「这一步完成了什么？一句话描述即可」
   - 等待用户确认后截取当前屏幕（调用 analyze_image）
   - 记录用户的一句话描述
3. 用户说"好了"后：
   - 汇总所有步骤：步骤描述 + 截图时间线 + 用户的一步一句描述
   - 推断意图（见下方"意图推断与确认"流程）
4. 关键约束：
   - 行为捕捉依赖用户配合提供一步一句描述，不能静默采集
   - 如果不支持实时截图（如 analyze_image 不可用），退化为纯文字描述模式
   - 文字模式下，请用户用 3-5 句话说明操作了哪些步骤
```

### 意图推断与确认

每一轮操作结束后，执行以下推断→确认循环：

```
1. 汇总行为序列：用户做了 A → B → C → D
2. 推断意图：用户可能在做 [X]，目的是 [Y]
3. 发现模糊点：步骤 B 和 C 之间的停顿 / 步骤 D 的重复执行
4. 生成确认问题（≤3 个）：
   - "我看到你在 [步骤X] 时 [行为]，你的目的是 [A] 还是 [B]？"
   - "这个操作的结果是你想要的吗？还是需要调整？"
5. 等待用户确认
6. 根据确认调整推断，继续下一轮或锁定意图
```

### 意图锁定 → Skill 规格书

当推断置信度 > 80% 且用户确认后，生成 **Skill 规格书**：

```markdown
## Skill 规格书

- 名称：[推测的 Skill 名称]
- 一句话定位：[这个 Skill 替用户省掉什么痛苦]
- 触发场景：[用户什么时候会用到它]
- 输入：[Skill 接收什么]
- 输出：[Skill 交付什么可见产物]
- 关键步骤：
  1. [从行为中提取的步骤1]
  2. [从行为中提取的步骤2]
  ...
- 用户特别偏好：
  - [从行为细节中发现的偏好，如习惯的快捷键、偏好的输出格式等]
- 用户确认的歧义点：
  - [澄清过的模糊需求及最终决定]
```

规格书展示给用户，确认后进入 Phase 2。

## Phase 1-B：识意——需求直译

> 适用于能清晰描述需求的用户。

1. 接收用户自然语言描述
2. 提取关键要素：做什么、输入什么、输出什么、什么时候用
3. 如有歧义或缺失项，用 ≤3 个问题确认
4. 生成 Skill 规格书，确认后进入 Phase 2

## Phase 2：铸胚——生成初版 Skill

### 生成内容

基于规格书，一次性生成：

1. **SKILL.md**：完整 Skill 定义文件，包含：
   - 规范的 frontmatter（name、description 含触发词、≤1024 字符）
   - 清晰的 Phase/Step 工作流
   - 每个步骤的输入/输出定义
   - 失败模式编码（if X 失败 → Y）
   - 检查点设计（🔴 CHECKPOINT / 🛑 STOP）
   - 反例黑名单（"不要做什么"）
   - Runtime 中立措辞（不绑定单一平台）

2. **test-prompts.json**（3 个测试 prompt）：
   ```json
   [
     {"id": 1, "prompt": "...", "expected": "..."},
     {"id": 2, "prompt": "...", "expected": "..."},
     {"id": 3, "prompt": "...", "expected": "..."}
   ]
   ```

3. **README.md** 初稿框架：
   - 一句话钩子
   - 触发方式（5-8 条真实触发语）
   - 快速开始
   - 可见产物展示
   - 安全边界

### 🔴 CHECKPOINT · 🛑 STOP：展示初版所有文件，等用户确认后进入 Phase 2.5 试刃。

---

## Phase 2.5：试刃——用户测试反馈循环

> 初版 Skill 不是终点，用户实测后的反馈才是真正的打磨起点。本阶段支持两种反馈渠道：自然语言描述问题，或通过操作演示期望的正确行为。反复循环直至用户满意，才进入 Phase 3 淬火。

### 试刃流程总览

```
铸胚完成（Phase 2）
      │
      ▼
┌─────────────────────────────────────────────┐
│  用户测试初版 Skill                           │
│  （在实际场景中触发 Skill，观察执行结果）        │
│                                              │
│  满意？── 是 ──→ 进入 Phase 3 淬火             │
│  否 ↓                                        │
│  发现问题 / 期望行为不符                       │
│      │                                       │
│      ├── 反馈方式 A：自然语言描述               │
│      │    "第三步不对，应该是先排序再筛选"        │
│      │    "输出格式我希望是 Markdown 表格"       │
│      │    "压缩比例太高了，画质损失太大"          │
│      │                                       │
│      └── 反馈方式 B：操作演示                    │
│           "你看我怎么做" → 用户操作演示           │
│           行为捕捉 → 意图识别 → 偏差分析           │
│                                              │
│      ▼                                       │
│  分析问题本质 + 对比用户期望 vs 当前 Skill 行为    │
│      │                                       │
│      ▼                                       │
│  生成修正方案（含备选方案，供用户选择）             │
│      │                                       │
│      ▼                                       │
│  用户确认方案 → 执行修正 → 更新 Skill 文件        │
│      │                                       │
│      └──→ 回到「用户测试初版 Skill」继续循环       │
└─────────────────────────────────────────────┘
```

### 反馈方式 A：自然语言描述

用户直接用自然语言指出问题。处理流程：

```
1. 接收用户的问题描述
2. 定位问题所属层面：
   - 行为层：Skill 执行的动作不对（步骤顺序错、遗漏操作、多余操作）
   - 输出层：Skill 产出的结果不对（格式错、内容错、质量不达标）
   - 交互层：Skill 的触发/确认/暂停逻辑不符合预期
   - 边界层：Skill 在特定条件下行为异常
3. 确认理解（≤2 个问题）：
   - "你是说 [复述问题]，对吗？"
   - "你期望的结果是 [A] 还是 [B]？"
4. 生成修正方案
```

### 反馈方式 B：操作演示（行为捕捉修正）

用户通过实际操作演示期望的正确行为。这是 Phase 1-A 行为捕捉能力的延续，但聚焦于"修正"场景。

#### 启动操作演示

告知用户："请操作一遍你期望的正确流程，完成后告诉我'好了'。你可以只操作需要修正的部分。"

#### 行为捕捉与偏差分析

```
1. 观察用户操作序列：用户做了 A' → B' → C' → D'
2. 回看当前 Skill 的行为序列：Skill 做了 A → B → C → D
3. 逐步骤对比偏差：
   - 步骤 1：用户做 [A']，Skill 做 [A] → 一致 / 不一致
   - 步骤 2：用户做 [B']，Skill 做 [B] → 一致 / 不一致
   ...
4. 识别偏差类型：
   - 步骤缺失：用户做了但 Skill 没做
   - 步骤多余：Skill 做了但用户没做
   - 步骤顺序错误：步骤相同但顺序不同
   - 参数差异：步骤相同但参数/配置不同
   - 交互差异：确认点、暂停点、输出格式不同
5. 生成偏差分析报告（展示给用户）
```

#### 偏差分析报告格式

```markdown
## 偏差分析报告（第 N 轮试刃）

### 你做的事情 vs Skill 做的事情

| 步骤 | 你的操作 | Skill 当前行为 | 偏差 |
|------|---------|---------------|------|
| 1 | 按修改日期排序 | 按文件名排序 | 🔴 参数差异 |
| 2 | 选中前 10 个文件 | 选中所有文件 | 🔴 参数差异 |
| 3 | 批量重命名为"报告_序号" | 移动文件到子目录 | 🔴 步骤不同 |
| - | （无此步骤） | 发送通知 | 🟡 步骤多余 |

### 推断的修正意图

你希望 Skill 做的是：...（归纳）

### 关键确认问题

- "步骤 3，你的意思是压缩前先预览，还是完全跳过压缩？"
```

### 修正方案生成

基于偏差分析，生成修正方案。**必须提供至少 2 个方案**（含备选），让用户选择：

```markdown
## 修正方案（第 N 轮试刃）

### 方案 A：严格按你的操作修正
- 改动：将 Skill 行为完全对齐你的操作序列
- 优点：完全符合你的习惯
- 风险：可能丢失其他人的通用性
- 改动范围：[列出要改的文件和段落]

### 方案 B：归纳你的操作意图，提供更优实现
- 改动：[归纳后的改进思路]
- 优点：[相比方案 A 的优势]
- 风险：[可能的风险]
- 改动范围：[列出要改的文件和段落]

### 其他可行方案（如适用）
- 如果行为演示中发现了可优化的冗余步骤，主动提出简化方案
- 如果用户的操作用到了外部工具组合，可建议将工具链整合进 Skill
- 如果用户的操作有潜在的性能/兼容性问题，提供替代技术路线

请选择方案，或告诉我需要混合/调整的部分。
```

### 执行修正

用户选择方案后：

```
1. 按选定方案编辑 SKILL.md（以及 README.md、scripts 等关联文件）
2. 更新 test-prompts.json，追加本轮发现的边界场景
3. 展示改动 diff
4. 请用户再次测试
5. 循环直到用户确认满意
```

### 修正回滚机制

试刃过程中出现错误修正、方向偏离或用户反悔时，支持**两级回滚**——回退到上一轮或上两轮的状态。

#### 回滚触发条件

以下任一用户表述自动触发回滚询问：

- "不对，改错了""还不如之前""越改越差""回退""撤销""恢复上一版"
- "回到上一步""回到之前那个版本""刚才那个版本更好"
- "改回去""不要这个改动了"

#### 回滚流程

```
1. 确认回滚目标：
   - "你想回滚到哪一轮的状态？"
   - 展示最近 3 轮的简要摘要（轮次号 + 改动摘要 + 用户当时确认状态）

2. 用户指定目标轮次 N：
   - 回滚到 trial-N 结束时的状态
   - 即：撤销 trial-(N+1) 及之后的所有修正

3. 执行回滚：
   方案 A（git 可用）：
     git log --oneline blade-trials/ → 找到 trial-N 对应的 commit
     git revert <trial-N+1 到 HEAD 的所有 commit>
   方案 B（git 不可用，文件备份）：
     从 blade-trials/trial-N-fix.diff 逆向应用
     或用 trial-N 时刻的 SKILL.md.bak.trial-N 覆盖当前文件

4. 回滚后操作：
   - 将回滚动作记录到 trial-log.tsv（action=rollback, from=N+1, to=N）
   - 展示回滚后的 Skill 当前状态
   - 询问："回滚完成，当前是第 N 轮修正后的状态。继续修正还是进入 Phase 3？"

5. 回滚后的下一轮编号：
   回滚到 trial-N 后，下一轮修正编号为 trial-(N+1)，覆盖原来的 trial-(N+1)
```

#### 回滚深度限制

- 最多回滚到 **trial-00**（Phase 2 铸胚完成时的初始状态）
- 每次回滚最大跨度：**2 轮**（如当前是 trial-05，最多回滚到 trial-03）
- 如需回滚更深（如从 trial-05 回到 trial-01），需分步执行并每步确认

#### 回滚保障：每轮自动快照

在执行修正前，自动保存当前 Skill 文件的快照：

```
每次进入「执行修正」步骤时：
1. 自动创建 SKILL.md.bak.trial-N（N 为当前轮次）
2. 自动创建 README.md.bak.trial-N（如存在）
3. 记录快照时间戳到 trial-log.tsv
```

这样即使 git 不可用，文件级快照也能保证回滚能力。

### 试刃循环退出条件

以下任一条件满足时退出循环，进入 Phase 3：

- 用户明确说"可以了""满意了""没问题了""进入下一步"
- 连续 2 轮用户没有提出新的修正需求
- 用户主动要求跳过后续修正（需确认）

### 试刃过程文件管理

每轮试刃的记录保存为独立文件，便于回溯：

```
[skill-name]/
├── blade-trials/              # 试刃记录目录
│   ├── trial-01-deviation.md  # 第 1 轮偏差分析报告
│   ├── trial-01-fix.diff      # 第 1 轮修正 diff
│   ├── trial-02-deviation.md  # 第 2 轮偏差分析报告
│   └── ...
└── trial-log.tsv              # 试刃日志（轮次、反馈方式、修正摘要、用户确认状态）
```

### 试刃反馈关键词触发

以下用户表述自动识别为试刃反馈，不需要用户显式说出"进入试刃"：

- 描述类："不对""这里有问题""应该是...""我希望...""改成..."
- 演示类："你看我怎么做""应该是这样""正确的做法是""看我操作""跟着我做"
- 修正类："改一下""调整一下""这里换一种方式""能不能..."

## Phase 3：淬火——Darwin 自进化循环

> 融合 Darwin Skill 的完整自进化机制。核心原则：评估 → 改进 → 实测验证 → 保留或回滚。

### 九维评分 Rubric（总分 100）

| # | 维度 | 权重 | 评分标准 |
|---|------|------|---------|
| 1 | Frontmatter 质量 | 7 | name 规范、description 含触发词、≤1024 字符、无空话尾巴 |
| 2 | 工作流清晰度 | 12 | 步骤明确可执行、有序号、每步有明确输入/输出 |
| 3 | 失败模式编码 | 12 | 显式 if-then fallback、有错误恢复路径 |
| 4 | 检查点设计 | 6 | 关键决策前有 🔴/STOP 标记，不是"建议"措辞 |
| 5 | 可执行具体性 | 17 | 无模糊措辞、有具体参数/格式/示例 |
| 6 | 资源整合度 | 4 | references/scripts/assets 引用正确、路径可达 |
| 7 | 整体架构 | 12 | 结构清晰、不冗余不遗漏、无 AI 腔废话 |
| 8 | 实测表现 | 23 | 用 test-prompts 跑实测，输出质量达标 |
| 9 | 反例与黑名单 | 7 | 有"不要做什么"清单、危险动作标注 |

**总分 = Σ(维度分 × 权重) / 10**，满分 100。

### 优化循环（默认 MAX_ROUNDS=3）

```
Phase 3.1 基线评估：
  - 结构评分：按维度 1-7 逐项打分
  - 效果评分：用独立子 agent 跑 test-prompts，对比带/不带 skill 的输出
  - 记录到 results.tsv
  - 🔴 CHECKPOINT：展示评分卡，等用户确认

Phase 3.2 优化循环（分数从低到高）：
  for each skill (或仅当前 skill):
    round = 0
    while round < MAX_ROUNDS:
      round += 1
      1. 诊断最低维度（注意 dim2/3/4 相关簇）
      2. 提出改进方案（改什么、为什么改、预期提升）
      3. 执行改进（编辑 SKILL.md）
      4. 重新评估（独立子 agent 评分）
      5. 决策：
         if 新总分 > 旧总分: keep，更新旧分
         if 连续2轮 Δ < 2: break（触顶信号）
         else: revert + break
      6. 记录到 results.tsv
      🔴 CHECKPOINT：每个 skill 优化完后暂停，展示 diff 和分数变化

Phase 3.3 探索性重写（按需）：
  当 hill-climbing 连续触顶时，提议一次从头重写。
  必须征得用户同意后才执行。
```

### Darwin 反例黑名单（锻造时禁止做的事）

| # | 反模式 | 替代做法 |
|---|--------|---------|
| 1 | 同 context 自评自改 | 必须 spawn 独立子 agent 评分 |
| 2 | 用 `git reset --hard` 回滚 | 用 `git revert HEAD` 保留追溯链 |
| 3 | 为凑分增冗余 | 触顶信号 → break，见好就收 |
| 4 | 跳过 test-prompts 直接评分 | Phase 0 强制设计 3 个 prompts |
| 5 | 轮内改多个维度 | 每轮 1 个维度 |
| 6 | dry_run 比例 > 30% | 强制至少 1 个 full_test |
| 7 | 静默跳过异常 | 异常先告知用户再处理 |

#### 回滚触发

以下表述自动触发回滚：

- "这个 README 还不如之前""展示效果没之前好"
- "把 README/证书/截图 改回去""退回打磨前"

#### 回滚方式

```

  - SKILL.md.bak.phase4-blade
  - README.md.bak.phase4-blade

回滚粒度（三级）：
  1. 微回滚：撤销最近一次 README 或 showcase 的单项修改
     → 从 blade-trials/ 目录找到对应的修改 diff，逆向应用
  2. 中回滚：回退到 Phase 4 开始时的快照（SKILL.md + README.md）
     → 用 .bak.phase4-blade 文件覆盖当前文件
  3. 深回滚：回退到 Phase 3 淬火完成时的状态

     → 需要用户二次确认

回滚后：
  - 记录日志到 trial-log.tsv（action=blade-rollback, level=微/中/深）
  - 展示回滚前后的差异

```

#### 与试刃回滚的区别

| 维度 | Phase 2.5 试刃回滚 | Phase 4 开刃回滚 |
|------|-------------------|-----------------|
| 回滚对象 | SKILL.md 核心逻辑 + scripts | README + showcase + 出师证书 |
| 回滚跨度 | 最多 2 轮试刃修正 | 最多回到淬火完成时的状态 |
| 触发场景 | 功能逻辑修正出错 | 展示/文档打磨方向偏离 |
| 回滚粒度 | 按轮次 | 按修改项（微/中/深三级） |

## 异常与边界条件

| 场景 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|---------|-----------|
| 行为捕捉无有效输入 | 用户 30 秒无操作 | 提示"请开始操作，完成后告诉我'好了'" | 切换到需求描述路径 Phase 1-B，并告知切换原因 |
| 意图推断置信度低 | 行为模式不一致，多个意图可能 | 列出 2-3 个可能的意图让用户单选 | 请用户用一句话描述目的，转为 Phase 1-B 混合模式 |
| 用户中途打断 | 行为捕捉过程中用户说"等等""不对" | 丢弃当前轮推断，询问"重新开始还是从某个步骤继续？" | 保留已确认的历史轮次推断，从打断点重新捕捉 |
| 试刃用户跳过反馈 | 用户说"没问题"但 test-prompts 显示边界异常 | 主动提示："我注意到 XX 场景下可能会有 YY 问题，要不要看一下？" | 如用户坚持跳过，记录 boundary-warning 到 trial-log.tsv 并继续 |
| 试刃循环超过 5 轮 | 用户反复提出修正但每次方向不同 | 暂停循环，展示 5 轮修正方向变化趋势 | 建议回到 Phase 1 重新锁定意图，或保留最新版本进入 Phase 3 |
| 试刃偏差无法识别 | 用户操作与 Skill 行为表面一致但结果不同 | 逐步骤展示中间状态和输出，对比差异 | 请用户指出具体分歧点，或录制屏幕操作供详细比对 |
| 依赖安装失败 | pip/npm/brew 命令报错 | 展示错误信息，尝试替代安装方式（如 pip→conda, npm→yarn） | 提供手动安装命令清单，标记依赖为「需手动安装」并继续封装 |
| 子 Agent 不可用 | 评分阶段 dispatch_task 失败 | 退化为干跑验证，按 dim1-7 做结构评分 | 标注 dry_run，在 results.tsv 的 eval_mode 列写 dry_run，提示用户后续补实测 |
| Skill 体积超限 | 优化后 SKILL.md > 原 150% | 检查是否有重复段落或可外移到 references/ 的内容 | 拒绝提交，展示冗余分析报告，等用户确认精简方案 |
| 平台封装缺少必要字段 | 目标平台必填字段缺失 | 提示用户补充缺失字段的具体内容 | 使用默认值填充并标注「默认值」，在 README 中注明需手动复核的字段 |

## 约束规则

1. **不改变核心功能**：淬火只优化"怎么写"和"怎么执行"，不改"做什么"
2. **不引入非必要依赖**：每个新增依赖都要有明确理由
3. **每轮只改一个维度**：避免多变量导致无法归因
4. **Runtime 中立**：Skill 必须能在 Claude Code、Codex、Cursor、OpenClaw、Hermes 等 runtime 运行
5. **可回滚**：所有改动可追溯，不用破坏性回滚
6. **评分独立**：效果维度必须用独立子 Agent，不能自评自改
7. **隐私保护**：不泄露 API key、token、cookie、私有路径、真实账号信息
8. **人在回路**：关键决策点必须暂停等用户确认

| # | 不要做 | 因为 | 应该做 |
|---|--------|------|--------|
| 1 | 用户说不清需求时直接放弃 | 行为捕捉就是为这个场景设计的 | 切换到 Phase 1-A |
| 2 | 跳过行为确认直接生成 Skill | 推断可能完全错误 | 每轮行为后必须确认 |
| 3 | 跳过试刃直接进淬火 | 用户实际需求可能和初版有偏差 | 完整走 Phase 2.5 直到用户满意 |
| 4 | 试刃时只给一个修正方案 | 用户需要选择权 | 必须提供 ≥2 个方案 |
| 5 | 试刃修正后不记录 trail | 无法回溯修正历史 | 每轮保存 deviation 报告和 diff |
| 6 | 淬火后不亮活 | 没有 README/showcase 无法传播 | 完整走 Phase 4 |
| 7 | 亮活后不封箱 | 用户还得手动整理发布 | 完整走 Phase 5 |
| 8 | 自评自改 | LLM-as-judge 准确率仅 46.4% | 必须独立子 Agent 评分 |
| 9 | 把依赖安装失败静默跳过 | Skill 装不上等于废品 | 展示错误并给手动方案 |
| 10 | 为凑分堆冗余 | 更长 ≠ 更好 | 触顶即停 |
| 11 | 泄露凭据或私有路径 | 安全底线 | Phase 4/5 强制脱敏检查 |
| 12 | 把 Runtime 写死 | 跨平台是公共 Skill 的基本要求 | Runtime 中立措辞 |
| 13 | 用户说"不对"时只改表面措辞 | 可能底层逻辑有偏差 | 深入偏差分析，定位问题层面 |
| 14 | 回滚时用破坏性方式（reset --hard） | 丢失可追溯历史 | git revert 或文件快照覆盖，保留日志 |
| 15 | 回滚后不告知用户当前状态 | 用户不知道自己在哪一轮 | 必须展示"已回滚到 trial-N，当前状态是..." |
